# AI Collaboration Card — 校园轻集市

本文档记录在"校园轻集市"项目开发过程中使用 AI Coding 工具的协作过程，包括各阶段使用的典型 Prompt、AI 输出内容、人工修改和最终验证结果。

---

## 一、AI 使用总览

| 开发阶段 | AI 辅助内容 | 人工承担内容 |
|----------|------------|------------|
| Day1 | 项目结构分析、技术栈梳理 | 业务场景理解、页面清单设计 |
| Day2 | 页面骨架代码生成、路由配置、导航组件 | 路由结构设计、Element Plus 配置调通 |
| Day3 | Mock 数据生成、API 封装、列表渲染模板 | 数据结构设计、接口契约定义 |
| Day4 | 表单模板生成、校验逻辑编写 | 四类业务字段设计、表单交互流程 |
| Day5 | Pinia Store 结构代码、收藏逻辑 | Store 划分决策、状态边界判断 |
| Day6 | 组件抽取、搜索/错误状态代码、样式 | 优化范围判断、AI 建议筛选 |
| Day7 | README 整理、检测报告、证据归档 | 验收标准判断、展示内容准备 |

## 二、各阶段协作详情

### Day1 — 项目启动与业务梳理

- **Prompt：** "帮我分析 Vue3 校园集市项目：1. 项目目录结构；2. 解释 main.ts；3. 说明 router 作用；4. 梳理项目技术栈。"
- **AI 输出：** 逐项分析了目录分层、入口文件作用、路由原理和技术栈组成
- **人工修改：** AI 快速理清项目分层，看懂项目启动流程与路由原理
- **最终验证：** AI 可辅助代码解析，但业务思考、项目开发仍需自主完成

### Day2 — 页面骨架与路由

- **Prompt：** "帮我生成 Vue3 校园集市 7 个页面骨架，全套路由代码，Element Plus 导航栏完整代码。"
- **AI 输出：** 基础页面模板、标准路由数组、原生导航代码
- **人工修改：** 修正路由中英文引号错误、给详情路由增加动态 id 参数、替换导航为 Element Plus 菜单
- **最终验证：** 全部页面可正常跳转，导航样式正常，无报错

### Day3 — Mock 数据与列表渲染

- **Prompt：** "为二手交易、失物招领、拼单搭子、跑腿委托四个业务模块各生成 6-8 条 JSON 数据，同时生成对应的 TypeScript 接口类型和 Axios 请求封装。"
- **AI 输出：** 完整的 db.json 数据、API 模块代码、列表页面 `v-for` 渲染模板
- **人工修改：** 统一字段命名风格（中文 key 改英文）、调整数据结构使各模块字段对齐业务含义
- **最终验证：** `pnpm mock` 启动成功，四个列表页面均能正常加载数据

### Day4 — 发布表单与数据新增

- **Prompt：** "为四类业务生成发布表单，每类有不同字段，支持表单校验，提交后通过 Axios POST 到 JSON Server，成功后跳转到对应列表页。"
- **AI 输出：** PublishView.vue 完整模板和脚本、四类业务字段的条件渲染、validate 函数
- **人工修改：** 发布人信息从硬编码改为读取 userStore、调整表单 UI 布局、优化校验提示文案、添加 ElMessage 反馈
- **最终验证：** 四类业务均可成功提交，数据出现在对应列表页

### Day5 — 状态管理

- **Prompt：** "使用 Pinia 创建用户 Store 和收藏 Store，用户 Store 管理模拟用户信息，收藏 Store 支持 add/remove/toggle/isFavorited 操作。"
- **AI 输出：** 两个 Store 的完整 Composition API 代码、UserCenterView 收藏展示模板
- **人工修改：** 确认 Store 不存放页面级临时数据（表单、loading）、调整 publisherContact 格式、审查命名清晰度
- **最终验证：** 导航栏显示用户信息，收藏功能跨页面正常，用户中心数据完整

### Day6 — 交互优化

- **Prompt：** "为所有列表页面添加 loading/empty/error 三态处理，抽取 LoadingState/ErrorState/SearchBar 组件，为失物招领/拼单搭子/跑腿委托添加收藏按钮。"
- **AI 输出：** 三个可复用组件、五个页面的状态逻辑重构、搜索功能实现
- **人工修改：** 拒绝了 debounce 建议（数据量小不需要）、拒绝了高级搜索面板（超出范围）、拒绝了额外 Toast 库（已有 ElMessage）
- **最终验证：** 构建通过，三态完整，搜索可用，收藏全页面覆盖

## 三、AI 生成内容的质量观察

### AI 表现较好的方面

- **重复性代码生成：** 组件模板、样式代码、API 封装等机械性工作
- **语法规范性：** TypeScript 类型定义、Vue 3 Composition API 写法
- **工具链配置：** Vite、ESLint、TypeScript 等配置文件
- **批量数据生成：** Mock 数据的结构化输出

### AI 容易出现问题的地方

- **路由路径字符串：** 中文引号与英文引号混用，导致语法错误
- **过度设计：** 倾向于添加 debounce、高级搜索、登录鉴权等超出范围的功能
- **上下文丢失：** 在多个文件同时修改时，可能遗漏部分文件的状态更新
- **业务逻辑判断：** 不熟悉校园场景的具体需求，需要人工调整

### 人工必须介入的判断

- **架构决策：** 哪些数据放入 Store、哪些保留在组件内
- **范围控制：** 拒绝超出实训范围的功能建议
- **命名规范：** 确保中文业务术语使用一致
- **交互细节：** 按钮位置、动画效果、提示文案
- **最终验收：** 确认每个功能的实际表现符合预期

## 四、AI 协作反思

### 效率提升

使用 AI Coding 工具在以下方面显著提升了开发效率：
- 省去了大量的模板代码编写时间（页面骨架、样式、API 封装）
- 减少了查阅文档的频率（API 语法、组件用法可即时生成）
- 加速了 bug 定位（错误信息直接交给 AI 分析）

### 能力要求变化

使用 AI 后，对开发者的能力要求发生了变化：
- **阅读代码 > 编写代码：** 需要快速理解 AI 生成的代码并判断是否正确
- **架构判断 > 实现细节：** AI 能写具体代码，但无法做架构决策
- **问题定位 > 问题描述：** 需要精确描述问题，而不是直接写解决方案
- **审查能力 > 创造能力：** 从"自己想怎么写"变为"判断 AI 写得对不对"

### 后续注意事项

- 小步提交，每完成一个小功能就提交，避免 AI 一次性修改过多文件
- 每次 AI 修改后必须运行项目验证，不能只靠看代码确认
- 明确告诉 AI"不要做什么"比只告诉它"要做什么"更重要
- AI 的优化建议需要主动筛选，不是所有建议都应该采纳
